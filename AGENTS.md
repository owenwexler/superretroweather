# DESIGN AND ARCHITECTURAL PRINCIPLES
1. All user interface must be mobile-first and mobile-responsive, always.  If making a web app, assume an equivalent native mobile app either exists already or is part of a near-future roadmap, and design all user interface around this eventuality.
2. All user interface should be clean and aesthetically pleasing, following modern design principles.
3. On the web, we want a fast, snappy, native-app-like user experience as much as possible.
4. User feedback is essential for any blocking or async operation.  The user should never click a button to make something happen then wonder if it's happening.  It should also never be possible to click a submit button multiple times while a form submission or other such operation is in flight.  Loading states, disabled buttons, top loaders, loading... dialogs, any kind of user feedback is critical.
5. Following the aspiration for a snappy native-app-like user experience, optimistic updates should be used as much as possible on both web and mobile to provide the appearance of instant results to the user.  Optimistically updated elements must nonetheless be distinct from stable UI elements, usually through a pulsing animation and/or a "posting..." message, usually controlled by an "isOptimistic" boolean flag on the element's underlying data object.  Where optimistic updates are not possible, user feedback should be utilized as described above.
6. Proper rollback on server error must be in place for any optimistic updates.  Never leave the optimistic results of an errored async function in the UI and always inform the user of an error via feedback in the UI.
7.  We expect that our user base will be accessing our apps from all manner of internet speeds and devices.  We should always optimize with slow Internet and low-end phones in mind as much as possible.
8.  To that end, graphics and images load slowly.  We should use images as sparingly as possible and prefer vector images when we do use images, as much as possible.
9.  Native mobile apps should be local-first and offline-first as a rule.

## DIRECTORY RULES 
 
1. Components are never placed in `app/` directories.  `app/` contains only pages, layouts, loading states, error boundaries, and API routes.
2. All server-only code (database queries, server actions, service functions) lives in `server/`.  Never import from `server/` in a client component.
3. Zod schemas in `schemas/` are isomorphic — they may be imported from both client components and server code.  Schemas that must be server-only (e.g. those that import server-only libraries) belong in `server/` instead.
4. `lib/` is for pure functions and constants with no framework dependencies.  Code that depends on Next.js, React, or the database belongs in a more specific directory.
5. `typedefs/` contains only type and interface declarations — no runtime logic.  
6. One Drizzle migration directory will be generated at the project root by `drizzle-kit` — do not modify its contents manually.
7. `.env` is never committed.  `.env.example` must be updated every time a new environment variable is introduced.

# CODE STYLE
## If it is necessary ay any time to not conform to these code style guidelines, document any reasons for doing so via code comments as a rule.
1.  There must be semi-colons at the end of every line of code, unless it is not possible.  Files generated by ShadCN, TanStack CLI etc. with no semi-colons at the end of lines must be modified to have semi-colons at the end of lines as much as possible. WRONG: ```console.log('Hello World!')``` RIGHT: ```console.log('Hello World!');```
2.  Use double quotes for Tailwind classes in JSX, and static props in JSX.  Use single quotes everywhere else.
3.  Use spaces instead of tabs.  Two spaces per press of the tab button.
4.  Use arrow functions as much as possible, do not use function() unless it is absolutely necessary.  WRONG: ```function someFunction() { logic };``` RIGHT: ```export const someFunction = () => { logic };```
5.  Within a function, group related lines of code together and use whitespace to separate unrelated lines of code.

6.  Use interface instead of type for complex object types and type for union types and Drizzle type inferences from schema
```
export interface User {
  id: string;
  email: string;
}

export type UnionType = 'some string' | 'some other string';

export type Todo = inferSelect<typeof todos>;
```

7.  Avoid the use of ```var``` to declare variables, ```var``` has been outdated for 10 years now. 

8.  Use ```const``` to declare variables whenever possible, or ```let``` if a variable must be mutable.  Avoid making variables mutable as much as possible.

9.  Use async/await as much as possible for async functions.  Do not use .then or .catch unless absolutely necessary and document the reason it must be used via code comments if so.

10. Give every condition in a ternary expression its own line
WRONG:
```
const bool = someCondition ? doSomething() : doSomethingElse();
```

RIGHT:
```
const bool =
    someCondition
    ?
      doSomething()
    :
      doSomethingElse();
```

11.  Write if-else statements as follows:
```
if (someCondition) {
  doThing();
} else if (someOtherCondition) {
  doOtherThing();
} else {
  doDefaultThing();
}
```

12.  Export functions at declaration, e.g. ```export const someFunction = { logic };```

13.  Organize type imports separately from regular imports and always give type imports from a library from which other imports are made their own line.
WRONG:
```
import { success, failure, errors, type ErrorType } from "dataerror";
```

RIGHT:
```
import { success, failure, errors } from "dataerror";
import type { ErrorType } from "dataerror";
```

14.  Use generic types to make helper functions reusable as much as possible.

15.  Make "~" the alias import path regardless of framework, where applicable, unless the framework forces a specific alias path in which case use the framework's required alias path.

16.  When chaining functions (e.g. db.select().etc. with Drizzle or Supabase client), give each function on the chain its own line.
WRONG:
```
const todos = await db.select().from().todos();
```

RIGHT:
```
const todos = await 
  db.
  select().
  from().
  todos();
```

# ERROR HANDLING
## If it is necessary ay any time to not conform to these error handling guidelines, document any reasons for doing so via code comments as a rule.
1.  The proper sequence in error handling is: an error is reported to the frontend by the backend, and the frontend then reports the error to the user.
 
2.  Returned errors with strong typing are generally preferred over thrown errors.  This is because returned errors are strongly typed and in a predictable shape whereas thrown errors are frequently not.

3.  Wherever possible, use the dataerror library or the shape from dataerror library { data: T, error: { code: string; message: string; details: string; hint: string; } } (the latter hereinafter referred to as ErrorType) for any async function returns unless throwing an error in the frontend is necessary to take the user to an error page in a framework like NextJS.  dataerror uses async ```success``` and ```failure``` functions to return either success objects with ```{ data: T, error: null }``` or failure objects with ```{ data: null, error: ErrorType }```.  All returns from backend or server functions, whether successful or errored, should use these functions to return the expected shapes.  Keep in mind ```success``` and ```failure``` are async functions that require a type to be passed in. e.g. ```const result = await success<Todo[]>(todos);```.  The ```success``` and ```failure``` functions in dataerror return results in the ```DataErrorReturnType``` shape ```{ data: T | null, error: ErrorType | null }```

4. Avoid throwing errors as much as possible, unless throwing an error is necessary as described above, otherwise return errors as described above

5. When showing error messages or toasts in the frontend, use the error object's error code field (error.code), if this exists, to determine what error message should be shown.   

6. As a security rule, never reveal whether it is a user's email or password that is incorrect when an invalid credentials error (errors.invalidCredentials in the dataerror library) is returned.  Always use a generic "Incorrect e-mail or password" message.

# API ENDPOINT CONVENTIONS
- Server actions (NextJS) or server functions (TanStack Start) are preferred over API endpoints for almost all functions.
- Server actions/functions should be secured with CORS policies or auth checks if applicable.  Calling a server action separately via Postman or some other HTTP client should never be possible.
- If an API endpoint is needed, use the following conventions: ```/auth``` for anything involving auth, ```/admin``` for any seed route, test route, test cleanup route, or anything involving multiple file deletions, secured with a special admin key, and ```/todos``` for any routes needed related to Todos that can not be a server action or function, or if we are using a HTMX/Alpine based stack that has no server functions or actions.

# AUTH
1.  If our project uses a full BaaS such as Supabase or Clerk for auth, we will use the offline mock auth service [Mockabase](https://github.com/owenwexler/mockabase) which mocks auth well enough for a development and testing environment on a local machine.  The reason to use Mockabase in development and testing is that it produces consistent results for testing, which live Supabase or Clerk responses do not always do. 2.  The latest version of the Mockabase client (available on the public Mockabase repo) should be used for any Mockabase functions.  Calling the underlying Mockabase API routes is not recommended.
3.  If using Better Auth, we just use Better Auth with our local database for development, testing, and production.
4.  A library such as Better Auth or a BaaS like Supabase or Clerk will address most security concerns related to auth (not storing tokens in localStorage, refreshing sessions regularly, etc.).  All authenticated server actions and API endpoints should have auth checks at the action/route level.

# BRANCHES
Our branches follow this convention
**main** - the top-level development branch, where code is verified and tests run.  Prefer not to push new features to **main**, but put them in separate branches.  Under no circumstances should any auto-deploy infrastructure point to **main**.  **main** is where everything is verified.  Ensure all tests pass and everything works as expected.
**staging** - if a staging environment exists (usually not the case for greenfield apps in early release), then all code that has been verified as working on main should be pushed to staging first.  This is a direct parallel copy of the production infrastructure.  All autodeploys for staging environments are set to point to **staging**.  NEVER make direct commits on or pushes to the **staging** branch.  The workflow, once all changes on main are approved and verified working, is to check out to the staging branch with ```git checkout```, run ```git merge main``` to merge **main** into the **staging** branch, then run ```git push``` to push the new merged branch to Github..
**production** - the branch that represents the live production environment seen and interacted with by users.  Any autodeploys on the production infrastructure point to this branch.  ONLY working, tested code verified in both development and staging (if applicable) is to be pushed to this branch.  NEVER make direct commits on or pushes to the **production** branch.  Once all changes are approved and verified working with tests passing, and verified to work in the staging environment if one exists, run ```git checkout production``` to check out the **production** branch, ```git merge main``` to merge the approved and tested changes from **main**, and ```git push``` to push the changes to the **production branch** where our hosting service's auto-deploy will pick up the changes and build a new production site.  
**IMPORTANT NOTE** After pushing to staging or production, it is critical to return to the **main** branch by typing ```git checkout main```.  Forgetting to do this and staying on the **staging** or **production** branch could result in accidentally pushing unauthorized changes to these branches that are very hard to roll back and create merge conflicts.
**Feature, chore, etc. branches** - new features, chores, and major migrations should be run on separate branches with the following convention: ```/feature/name-of-feature``` for features, ```chore/description-of-chore``` for chores, ```/migration/description-of-migration``` for migrations, etc.
